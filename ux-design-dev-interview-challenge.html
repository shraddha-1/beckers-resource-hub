Candidate Take-Home Assignment · Frontend / UX

Design a Content
Asset Portal

⏱ 4–6 hrs

Mid–Senior Level
Submit within 24 hrs of receiving

You're building a small, polished web experience for a media brand that lets visitors discover and sign up for content assets — think whitepapers, on-demand podcasts, and live and on-demand webinars. The focus of this exercise is entirely on frontend craft and UX thinking.

A real backend is provided and running — start it with npm run dev and it will serve data at http://localhost:3000. Wire your UI to the live API using fetch(). What we're evaluating is how you think about layout, interaction, component design, and the overall experience of browsing and converting.

What to build

Three pages, one cohesive experience

01

Homepage

Introduces the brand and surfaces the value of the content library. Drives visitors toward the asset listing.

02

Asset Listing

Displays the full catalog of available assets with filtering or sorting. Each asset is actionable.

03

Sign-Up Page

A focused conversion page for a single asset with a form, asset summary, and confirmation state.

Page requirements

Per-page specs

Page 1 Homepage First impression + navigation hub
  • Hero section with a headline, short value proposition, and a clear primary CTA that leads to the asset listing page.
  • Brand identity — invent a name, a simple mark or wordmark, and a consistent visual voice. Commit to it across all three pages.
  • Content teaser — show at least 2–3 featured or recent assets (title, type, short description). Pull from the live API rather than hardcoding.
  • Navigation — a functional nav bar with links to the listing page and a consistent layout that carries through all pages.
  • Footer — minimal but present. Can include nav links, a tagline, or a secondary CTA.
✦ Bonus Show a "personalized for you" or "recently viewed" section using localStorage. Seed from the live API on first visit.
Page 2 Asset Listing Catalog browse + discovery
  • Asset cards or list rows — each card should display the following fields from the API response:
    • name
    • assetType
    • sponsorName
    • description
    • executionDate (when present)
    Include a "Sign Up" or "Get Access" CTA that routes to the sign-up page for that asset's id.
  • Filter or sort controls — at minimum, allow filtering by assetType. The four valid values are:
    • Live Webinar
    • On-Demand Webinar
    • Whitepaper
    • on-demand podcast
    Multi-filter, keyword search, or sort-by-date are all encouraged.
  • Empty and loading states — show a loading indicator while GET /assets is in flight, and a friendly empty message when active filters return no results.
  • Pagination or infinite scroll — the API returns all assets in a single response. Avoid rendering all of them at once if it overwhelms the page.
  • Result count — display how many assets are currently showing, e.g. "Showing 8 of 10 assets".
✦ Bonus Add a debounced search bar that filters the API response client-side against name, description, and sponsorName. Or add a sort control by executionDate or lastModifiedDate.
Page 3 Sign-Up Page Single-asset conversion + form
  • Asset summary panel — load the asset via GET /assets/:id and display:
    • name
    • assetType (as a badge)
    • description
    • sponsorName
    • executionDate / expirationDate
    • speakers[] (if present)
  • Sign-up form — the following fields are all required by the API:
    • First name
    • Last name
    • Email
    • Job title
    • Company name
    Validate required fields and email format before submitting.
  • Submit behavior — POST to /assets/:id/signup with the body structure below. On a 201 response, show a confirmation state inline (no redirect) that surfaces the returned signupDate. 
    POST /assets/:id/signup
Content-Type: application/json

{
  "person": {
    "firstName": "...",
    "lastName": "...",
    "email": "...",
    "jobTitle": "...",
    "companyName": "..."
  }
}
  • Error handling — if the API returns a 4xx status, parse the { "error": "..." } response body and display the message inline without reloading the page.
  • Back navigation — provide an easy way to return to the listing. Preserving filter state is a plus.
✦ Bonus After a successful sign-up, fetch GET /assets and surface 2–3 "you might also like" assets by matching on assetType or sponsorName.
Provided for you

Live backend API

A real Express server is included in this repo. Run npm run dev to start it on http://localhost:3000. All responses use the same envelope: { "data": T } on success and { "error": "..." } on failure.

Returns all lead gen assets as an array inside data.

GET /assets/:id

Returns a single asset by its id. Returns 404 with { "error": "Asset not found" } when the id does not exist.

POST /assets/:id/signup

Registers a person for an asset. The operation is idempotent — submitting the same person and asset twice returns the same signup record.

Request body
{
  "person": {
    "firstName": "Jane",
    "lastName": "Smith",
    "jobTitle": "CFO",
    "companyName": "Acme Health",
    "email": "jane@acme.com"
  }
}
201 Response
{
  "data": {
    "id": "a3f9c2...",
    "assetId": "5af0e5...",
    "signupDate": "2026-05-11T...",
    "person": {
      "id": "7b1d44...",
      "firstName": "Jane",
      "lastName": "Smith",
      "email": "jane@acme.com",
      ...
    }
  }
}
400 is returned when the person field is missing from the request body. Optionally pass person.id to preserve an existing contact identity across signups.

A few notes from us

Tips & expectations

🎨

Bring a point of view

Invent a brand and commit to it. A distinctive, intentional aesthetic scores higher than a generic one — even if it's unconventional.

📐

Scope deliberately

3 polished pages beat 5 rough ones. If you're short on time, nail the listing page and sign-up — that's the critical path.

✍️

Use real content

Don't use "Lorem ipsum." The backend ships 10 real-looking assets. Let the actual names, descriptions, and sponsor names drive your UI copy.

🔧

Any stack is fine

React, Vue, Svelte, plain HTML/CSS/JS — use what you're fastest and most confident with. We're not evaluating framework choice.

💬

Document the why

A short README note about a tradeoff or design decision shows us how you think, and we weight that heavily in reviews.

🚀

Bonus ≠ required

The bonus features reward initiative but missing them won't hurt you. Focus on the core experience first.
